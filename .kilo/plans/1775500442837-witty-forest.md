# Plan: Introduce Contract Definition Segment

## Objective

Insert a new **Contract Definition Segment** between the existing Planning Segment and Implementation Segment. This segment takes what was previously Phase 5 (Interface & Test Definition), adds a new developer-led Scaffolding Review phase after it, and pushes all subsequent phases up by 2.

---

## New Workflow Structure

| Segment | Phase | Name |
|---|---|---|
| Planning | 1 | Planning |
| Planning | 2 | Exploration |
| Planning | 3 | Design Interview |
| Planning | 4 | Planning Wrap-up |
| **Contract Definition** | **5** | **Interface & Test Definition** *(moved, unchanged)* |
| **Contract Definition** | **6** | **Interface & Test Review** *(new)* |
| Implementation | 7 | Prioritisation *(was 6)* |
| Implementation | 8 | Implementation *(was 7)* |
| Review | 9 | Manual Review *(was 8)* |
| Review | 10 | AI Code Review *(was 9)* |
| Review | 11 | Commit *(was 10)* |

---

## Changes Required

### 1. `rules/workflow.md`

- Add a new `### Contract Definition Segment` heading after Phase 4 and before the Implementation Segment.
- Move **Phase 5 — Interface & Test Definition** under this new segment heading (text is unchanged).
- Add **Phase 6 — Interface & Test Review** immediately after Phase 5 under Contract Definition:
  - Developer-led review of the scaffolded interfaces and tests
  - Developer reads through the interfaces and tests and compares them against the feature spec
  - Developer makes any edits needed to correct or improve them
  - AI assists on request (e.g., "does this interface cover this part of the spec?")
  - No implementation code is written in this phase
  - **Checkpoint:** After the review, the scaffold is considered locked. Any changes from this point require revisiting the spec.
- Renumber all phases in the Implementation and Review segments (+2 each):
  - Prioritisation: Phase 6 → Phase 7
  - Implementation: Phase 7 → Phase 8
  - Manual Review: Phase 8 → Phase 9
  - AI Code Review: Phase 9 → Phase 10
  - Commit: Phase 10 → Phase 11
- Update the Implementation Segment heading to cover Phases 7–8.
- Update the Review Segment heading to cover Phases 9–11.
- Update the `implementation_tasks.md` format reference (if it references phase numbers) — currently it does not, so no change needed there.

### 2. `skills/planning/SKILL.md`

- Phase range reference `(Phases 1–4)` — **no change needed**, Planning Segment is unaffected.

### 3. `skills/implementation/SKILL.md`

- Update description and body: `Phases 5–7` → `Phases 7–8`.

### 4. `skills/review/SKILL.md`

- Update description and body: `Phases 8–10` → `Phases 9–11`.

### 5. New skill: `skills/contract-definition/SKILL.md`

- Create a new skill file for the Contract Definition Segment.
- Name: `Contract Definition`
- Description: `Start and facilitate the contract definition segment of the development workflow (Phases 5–6).`
- Instructions: Guide through Phases 5–6 in order; Phase 5 produces the scaffold, Phase 6 (Interface & Test Review) is a developer-led review with AI assisting on request.
